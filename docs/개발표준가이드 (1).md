## 1. 프로젝트 개요

- Frontend Repository는 **1개**로 운영한다.
- 개발 인원은 **10명**이다.
- Framework는 **Next.js**를 사용한다.
- State Management는 **Redux + Redux-Saga**를 사용한다.
- 협업 도구는 **GitHub, Jira, SourceTree**를 사용한다.
- Backend는 **MSA** 구조이다.
- 모든 팀원이 하나의 Repository에서 동시에 작업하므로, 아래 규칙을 반드시 준수한다.
  - 브랜치 전략
  - 공통 파일 관리 규칙
  - 코드 작성 규칙
  - PR / Merge 규칙

## 2. 시스템 아키텍처 개요

### 2.1 전체 구성

```text
[Browser]
   │  HTTPS
[Next.js Frontend] ── (SSR/CSR, 단일 Repository)
   │  REST API (JSON)
[Spring Boot MSA 서비스군] ── 서비스(도메인)별 독립 배포
   │  JPA / MyBatis 등
[Oracle DB] ── 서비스별 스키마/테이블 분리 원칙
```

- Frontend는 Next.js 단일 Repository로 운영하되, 내부적으로는 `{service}` 단위(환자, 접수, 검사/영상 등)로 화면·상태·API 로직을 분리한다.
- Backend는 서비스(도메인) 단위로 독립 배포되는 MSA 구조이며, 서비스 간 직접 DB 참조를 하지 않고 REST API로만 통신하는 것을 원칙으로 한다.
- Frontend는 각 서비스의 API를 개별 호출하며, 별도의 BFF(Backend For Frontend) 계층을 두지 않는다. (필요 시 프론트 리더와 협의 후 도입)

### 2.2 계층별 책임 (Frontend)

| 계층 | 위치 | 책임 |
| --- | --- | --- |
| Presentation | `app/`, `components/{service}` | 화면 렌더링, 사용자 입력 처리 |
| State / Side-effect | `features/{service}/slice.ts`, `saga.ts` | 상태 관리, 비동기 처리(API 호출) |
| API | `features/{service}/api.ts` | 서비스별 REST API 호출 함수 |
| 공통 | `lib/`, `constants/`, `types/`, `hooks/` | axios instance, 공통 타입/상수/훅 |

- 화면(Presentation)은 상태나 API를 직접 다루지 않고, `features/{service}`가 제공하는 selector/action을 통해서만 데이터를 주고받는다.
- 서비스 간 상태 직접 참조는 금지한다. (9.1 참고)

### 2.3 적용 기술 및 도구 표준

| 구분 | 항목 | 내용 | 용도 |
| --- | --- | --- | --- |
| 런타임 | Node.js | LTS 버전 고정 (`.nvmrc` 관리) | 개발/빌드 환경 통일 |
| Framework | Next.js | App Router 사용 | Frontend 프레임워크 |
| 패키지 관리 | pnpm | Lockfile 커밋 필수 | 의존성 버전 고정 |
| 상태 관리 | Redux Toolkit + Redux-Saga | slice/saga 단위 분리 | 전역 상태 / 비동기 처리 |
| HTTP 통신 | axios | 공통 instance + interceptor | REST API 호출 |
| 코드 품질 | ESLint / Prettier | 팀 공통 설정 사용 | 정적 분석, 포맷 통일 |
| 형상관리 | GitHub | PR 기반 협업 | 소스 버전 관리 |
| 이슈관리 | Jira | 브랜치/Commit/PR과 연동 | 작업 추적 |
| 로컬 Git 도구 | SourceTree | GUI 기반 브랜치/Merge 확인 | 형상관리 보조 도구 |
| CI/CD | Jenkins | develop/main Merge 시 자동 빌드 | 빌드/배포 자동화 |
| 컨테이너 | Docker | 서비스별 이미지 빌드 | 배포 표준화 |
| 배포 환경 | Nginx (사내망) | 정적 리소스 서빙, 리버스 프록시 | 운영 배포 |
| Backend | Spring Boot (MSA) | 서비스별 독립 배포 | REST API 서버 |

| 모니터링 | Sentry (또는 팀 협의 도구) | 운영 환경 에러 수집 | 장애 대응 |

> 위 표의 구체적 버전(Node LTS 버전 등)은 팀 협의 후 `.nvmrc`, `package.json`의 `engines` 필드에 고정한다.

## 3. 브랜치 전략

### 3.1 기본 브랜치

- `main`
  - 운영(Release) 브랜치
  - 배포 가능한 안정 코드만 유지
- `develop`
  - 통합 개발 브랜치
  - 서비스 브랜치 Merge 대상

### 3.2 서비스 브랜치

- 각 담당 서비스별 브랜치를 생성한다.
- 서비스 담당자는 자신의 서비스 브랜치에서 작업을 시작한다.


| 서비스   | 서비스 코드 | 브랜치명                  | 폴더명(camelCase) |
| ----- | ------ | --------------------- | --------------- |
| 환자    | PAT    | `service/patient`     | `patient`        |
| 접수    | RCP    | `service/reception`   | `reception`      |
| 수납/청구 | BIL    | `service/billing`     | `billing`        |
| 외래    | OPD    | `service/outpatient`  | `outpatient`     |
| 응급    | EMG    | `service/emergency`   | `emergency`      |
| 입원    | IPT    | `service/inpatient`   | `inpatient`      |
| 검사/영상 | LAB    | `service/lab-imaging` | `labImaging`     |
| 약국    | PHM    | `service/pharmacy`    | `pharmacy`       |
| 수술    | SUR    | `service/surgery`     | `surgery`        |
| 관리자   | ADM    | `service/admin`       | `admin`          |

> **서비스 코드**는 13장 코드 컨벤션, 15장 메시지/로그 코드 체계에서 동일하게 재사용한다.

### 3.3 Epic 브랜치

- 서비스 브랜치에서 생성한다.
- Jira Epic 단위로 생성한다.

**예시**

- `zp2-1-lab-specimen`
- `zp2-2-lab-result`
- `zp2-3-imaging-acquisition`
- `zp2-4-imaging-interpretation`
- `zp2-5-common`

### 3.4 Story 브랜치

- Epic 브랜치에서 생성한다.
- Jira Story / Task 단위로 생성한다.

**예시**

- `feature/lab-specimen/zp2-12-lab-order`
- `feature/lab-specimen/zp2-11-lab-schedule`
- `feature/lab-result/zp2-13-general-result`

### 3.5 브랜치 네이밍 규칙

- Service: `service/{service-name}`
- Epic: `{jira-epic-key}-{short-name}`
- Story: `feature/{epic-name}/{jira-story-key}-{short-name}`
- Bugfix: `bugfix/{jira-key}-{short-name}`
- Hotfix: `hotfix/{jira-key}-{short-name}`

### 3.6 로컬 브랜치 / 원격 브랜치 구분

- **로컬 브랜치**: 내 PC에서만 존재하는 작업 브랜치
- **원격 브랜치**: GitHub에 존재하는 브랜치 (`origin/...`)
- 로컬과 원격을 혼동하지 않는다.
- 작업 전후 반드시 원격 최신 상태를 확인한다.

### 3.7 일일 작업 흐름 (필수)

```text
Pull → 작업 → Commit → Pull → Push
```

1. **Pull** : 원격 최신 코드를 먼저 받는다
2. **작업** : 로컬에서 개발한다
3. **Commit** : 로컬에 변경 사항을 기록한다
4. **Pull** : Push 전 원격 변경을 다시 받아 충돌을 확인한다
5. **Push** : 원격 브랜치에 반영한다

- Push 전 Pull을 생략하지 않는다.
- `main`, `develop`, `service/*`에 force push 금지

### 3.8 Merge 순서

1. Story
2. Epic
3. Service
4. Develop
5. Main

- 여러 기능을 먼저 서비스 단위에서 통합한 뒤, 최종적으로 Develop에 반영한다.
- 순서를 건너뛰는 Merge는 금지한다.

## 4. 개발 진행 순서

1. Develop 최신 Pull
2. Service 브랜치 생성 / Checkout
3. Epic 브랜치 생성
4. Story 브랜치 생성
5. 개발 (`Pull → 작업 → Commit → Pull → Push`)
6. Story → Epic PR 생성 후 Merge
7. Epic 기능 통합 및 테스트
8. Epic → Service PR 생성 후 Merge
9. Service 통합 테스트
10. Service → Develop PR 생성 후 Merge
11. 전체 테스트
12. Release 시 Develop → Main Merge

## 5. 프론트 리더 역할

프론트엔드는 공통 파일이 많으므로, 프론트 리더가 아래 영역을 관리한다.

### 5.1 RootReducer 관리

- 각 팀원은 자신의 slice / reducer만 작성한다.
- RootReducer 등록은 프론트 리더가 담당한다.

```ts
combineReducers({
  patient,
  reception,
  billing,
  labImaging,
  pharmacy,
});
```

### 5.2 RootSaga 관리

- 각 팀원은 자신의 saga만 작성한다.
- RootSaga 등록은 프론트 리더만 수정한다.

```ts
yield all([
  patientSaga(),
  receptionSaga(),
  billingSaga(),
  labImagingSaga(),
]);
```

### 5.3 공통 Layout 관리

- App Layout
- Sidebar
- Header
- Footer
- Navigation
- Theme

### 5.4 Router / Page 관리

- Next.js `app/` 구조는 프론트 리더가 관리한다.
- 팀원은 자신의 화면(컴포넌트/feature)만 개발한다.
- 라우트 등록이 필요하면 프론트 리더에게 요청한다.

### 5.5 공통 Component 관리

- Button
- Modal
- Table
- Pagination
- Dialog
- Input
- DatePicker
- 공통 컴포넌트 수정은 **프론트 리더 승인 후** 진행한다.

### 5.6 공통 API 관리

- axios instance
- Interceptor
- Token 처리
- Error 처리

### 5.7 메뉴 관리

- Sidebar Menu
- Top Menu
- Breadcrumb

### 5.8 공통 타입 / 상수 관리

- `types/`
- `constants/`
- `hooks/` (공통 hooks)
- 중복 정의를 방지하기 위해 프론트 리더가 관리한다.

## 6. 팀원 수정 가능 영역

- 각 팀원은 **자신의 서비스 영역만** 수정한다.
- 서비스 영역 안에서는 자유롭게 개발한다.

**프론트 수정 위치**

- `components/{service}/` : 화면 컴포넌트
- `features/{service}/` : api / saga / slice / types / utils

**백엔드**

- `@Controller` 는 **각자 담당 서비스**에서 작성·관리한다.

---

## 7. 수정 권한 제한 파일

아래 파일은 여러 명이 동시에 수정하면 Merge 충돌이 자주 발생한다.  
가능하면 **프론트 리더만 수정**한다.

- `rootReducer.ts`
- `rootSaga.ts`
- `store.ts` / `configureStore.ts`
- `layout.tsx`
- `menu.ts`
- `router.ts`
- `axios.ts`
- `theme.ts`
- 공통 `constants.ts`

**공통 파일 수정이 필요한 경우**

1. Jira 티켓에 수정 사유 작성
2. 프론트 리더에게 요청
3. 승인 후 수정 또는 리더가 직접 반영
4. PR Review 필수

## 8. 권장 프로젝트 구조

```text
src
├── app                         # Next.js App Router (리더 관리)
│   └── {service}/              # 서비스
│       ├── page.tsx            # 서비스 진입 페이지
│       └── {story}/            # Story (기능 단위)
│           ├── page.tsx        # Story 진입 페이지 (필요 시)
│           ├── list/page.tsx
│           ├── register/page.tsx
│           ├── detail/[id]/page.tsx
│           └── update/[id]/page.tsx
├── components                  # UI 컴포넌트
│   ├── auth                    # 공통 인증 (리더 관리)
│   ├── layout                  # 공통 레이아웃 (리더 관리)
│   ├── sidebar                 # 공통 사이드바 (리더 관리)
│   └── {service}               # 서비스별 화면 (담당자 작성)
├── features                    # 서비스별 상태 + API 로직
│   └── {service}
│       ├── api.ts              # API 호출
│       ├── saga.ts             # Redux-Saga
│       ├── slice.ts            # Redux Slice
│       ├── types.ts            # 요청/응답/상태 타입
│       ├── messages.ts         # 서비스 메시지 코드 사전 (15장 참고)
│       └── utils.ts            # 서비스 전용 유틸
├── constants                   # 공통 상수 (리더 관리)
├── lib
│   ├── axios.ts                # 공통 axios instance (리더 관리)
│   └── logger.ts               # 공통 로거 유틸 (리더 관리, 15.4 참고)
├── store
│   ├── rootReducer.ts          # 리더 관리
│   ├── rootSaga.ts             # 리더 관리
│   └── store.ts                # 리더 관리
├── types                       # 공통 타입 (리더 관리)
└── utils                       # 공통 유틸
```

### 8.1 app 라우트 계층 (필수)

```text
app/{service}/{story}/{pageType}/page.tsx
```


| 계층           | 의미            | 예시                                        |
| ------------ | ------------- | ----------------------------------------- |
| `{service}`  | 서비스 단위        | `labImaging`, `reception`, `billing`      |
| `{story}`    | Story / 기능 단위 | `labSpecimen`, `labResult`, `reservation` |
| `{pageType}` | 화면 유형         | `list`, `register`, `detail`, `update`    |


**예시 URL / 폴더**

```text
app/labImaging/labSpecimen/list/page.tsx
→ /labImaging/labSpecimen/list

app/labImaging/labSpecimen/register/page.tsx
→ /labImaging/labSpecimen/register

app/labImaging/labSpecimen/detail/[id]/page.tsx
→ /labImaging/labSpecimen/detail/12

app/reception/reservation/detail/[id]/page.tsx
→ /reception/reservation/detail/12
```

- `list`, `register`, `detail`, `update` 는 Story 하위에 둔다.
- 서비스 바로 아래에 `list`를 두지 않는다. (예외: 리더 승인)

## 9. Git / Commit / PR 규칙

### 9.1 Commit Message 규칙

형식:

```text
(지라코드)
1. (내용) + (기능수정 | 기능추가 | 기능삭제)
2. (내용) + (기능수정 | 기능추가 | 기능삭제)
3. (내용) + (기능수정 | 기능추가 | 기능삭제)
```

- 첫 줄에 **Jira 코드**를 작성한다.
- 변경 내용은 **숫자(1, 2, 3...)로 구분**한다.
- 각 항목 끝에 작업 유형을 명시한다.
  - 기능추가
  - 기능수정
  - 기능삭제

**예시**

```text
ZP2-12
1. 검체 접수 목록 조회 화면 추가 기능추가
2. 검체 상태 필터 조건 수정 기능수정
3. 미사용 임시 버튼 제거 기능삭제
```

### 9.2 PR 생성 규칙

- 아래 단계마다 **반드시 PR을 생성**한다.
  - Story → Epic
  - Epic → Service
  - Service → Develop
- PR 제목: `[Jira-Key] 작업 요약`
- PR 본문에 아래를 포함한다.
  - 작업 목적
  - 변경 내용
  - 테스트 방법
  - 관련 Jira 링크
  - 공통 파일 수정 여부 (Yes/No)
- 자기 자신 Merge 금지 (핫픽스 제외, 리더 승인 시)

### 9.3 Merge 방식

- 모든 단계의 Merge 방식은 **Merge Commit** 으로 통일한다.


| 단계                | 방식    |
| ----------------- | ----- |
| Story → Epic      | Merge |
| Epic → Service    | Merge |
| Service → Develop | Merge |
| Develop → Main    | Merge |


- Squash / Rebase Merge는 사용하지 않는다.
- `main`, `develop`, `service/*`에 force push 금지

### 9.4 코드 리뷰

- 서비스 내부 코드: 서비스 담당자 + 1인 리뷰
- 공통 파일 변경: 프론트 리더 필수 리뷰
- Layout / Menu / Router / API / Store Root: 프론트 리더 Approve 필수

## 10. Redux / Redux-Saga 작성 규칙

- RootSaga 등록은 프론트 리더에게 요청한다.
- 파일 위치: `features/{service}/saga.ts`

### 10.4 Selector

- 상태 조회는 selector를 통해 수행한다.
- Component에서 `state.xxx.yyy.zzz` 깊은 접근을 지양한다.

## 11. API 호출 규칙

### 11.1 프론트 API 규칙

- 공통 axios instance만 사용한다.
- Token / Interceptor / 공통 Error 처리는 프론트 리더가 관리한다.
- 서비스 API 함수는 `features/{service}/api.ts`에 작성한다.
- API 함수 네이밍:
  - `getXxx`
  - `createXxx`
  - `updateXxx`
  - `deleteXxx`
- 응답 타입을 TypeScript로 명시한다.
- 하드코딩 URL을 금지하고, 환경변수 / constants를 사용한다.
- 환자명·처방의명 등 타 서비스가 소유한 정보는 저장된 값이 아니라 API 조회로 가져온다(14.1 참고). 목록 화면처럼 여러 건을 한 번에 표시할 때는 행마다 개별 호출하지 않고, 식별자 배열을 한 번에 넘겨 일괄 조회하는 배치 API를 사용한다.

### 11.2 REST API 방식


| 기능  | 방식       |
| --- | -------- |
| 조회  | `GET`    |
| 등록  | `POST`   |
| 수정  | `PUT`    |
| 삭제  | `DELETE` |


### 11.3 응답 포맷 (JSON 통일)

```json
{
  "code": 200,
  "message": "SUCCESS",
  "data": {}
}
```

- `code` : 결과 코드 (HTTP status 또는 서비스 정의 코드)
- `message` : 결과 메시지 — **사용자 노출용 메시지는 15장 메시지 코드 체계를 따른다.**
- `data` : 실제 응답 데이터

### 11.4 백엔드 Controller

- `@Controller` / `@RestController` 는 **각자 담당 서비스**에서 작성한다.
- 서비스 간 Controller를 임의로 수정하지 않는다.

### 11.5 예외 처리

- 서버 예외는 공통 Exception Handler로 일괄 처리한다.
- 프론트는 공통 axios interceptor에서 `code` / `message`를 기준으로 에러를 처리한다.
- 사용자에게 노출되는 에러 메시지와 개발자용 로그는 분리한다. (15.1 참고)

## 14. DB 규칙

### 14.1 기본 규칙

- **DBMS: Oracle 단일 사용을 원칙으로 한다.** 서비스/환경(개발·운영)에 관계없이 타 DBMS(MySQL, PostgreSQL, MSSQL 등)는 사용하지 않는다. 이후 모든 데이터타입 표기(14.2, 14.3)는 Oracle 기준이다.
- **타 서비스(도메인)가 소유한 데이터는 스냅샷 컬럼으로 저장하지 않는다.** 환자명, 처방의명, 담당자명 등 다른 서비스가 원본을 소유한 정보는 트랜잭션 테이블에 값으로 복제해 두지 않고, 참조 식별자(`patient_no`, `physician_no`, `received_by_id` 등)만 컬럼으로 보유한다. 표시가 필요한 시점에 해당 서비스 API를 호출해 조회한다. (2026-07-10 팀 협의로 스냅샷 방식 폐지, API 조회로 일원화)
  - 목록 화면 등에서 매 행마다 API를 호출하면 N+1 호출 문제가 발생할 수 있으므로, 백엔드에 식별자 배열을 한 번에 넘겨 이름을 일괄 조회하는 배치 조회 API 제공을 요청한다. (11.1 참고)
  - 다만 동의서 서명자명(`signed_by`)처럼 **다른 서비스의 원본이 아니라 그 화면에서 직접 입력·확정된 값**은 스냅샷이 아니라 원본 데이터이므로 그대로 저장한다.
- 테이블명: `UPPER_SNAKE_CASE` (대문자 + 언더바)
  - 예: `SURGERY`, `OPERATING_ROOM`
- 컬럼명: `lower_snake_case` (소문자 + 언더바)
  - 예: `surgery_id`, `scheduled_dt`
- PK: `{테이블명}_id`
- FK: `{참조테이블명}_id`
- 모든 테이블에 `created_at`, `updated_at` 공통 포함

> DB 규칙은 백엔드 소관이지만, 프론트에서 API 응답 필드명을 매핑할 때 참고한다.

### 14.2 컬럼 접미사 규칙

| 접미사 | 의미 | 데이터타입(Oracle) | 예시 |
| --- | --- | --- | --- |
| `_id` | 식별자(PK/FK) | VARCHAR2(36) 또는 NUMBER | `patient_id`, `lab_order_id` |
| `_no` | 채번된 업무 번호 | VARCHAR2(20) | `reception_no`, `lab_order_no` |
| `_cd` / `_code` | 코드성 컬럼(코드마스터 참조) | VARCHAR2(코드 길이) | `system_code`, `order_status` |
| `_yn` | 여부(Boolean) | CHAR(1), `'Y'`/`'N'`만 허용 | `urgency_yn`, `use_yn` |
| `_dt` | 날짜 | DATE | `order_dt` |
| `_at` | 일시(타임스탬프) | TIMESTAMP | `received_at`, `created_at` |
| `_nm` / `_name` | 명칭 | VARCHAR2(100) | `patient_name` |

### 14.3 데이터타입 표준

| 컬럼 종류 | 데이터타입(Oracle) | 길이 기준 | 비고 |
| --- | --- | --- | --- |
| 개수 / 수량 / 금액 | `NUMBER` | 소수 필요 시 `NUMBER(p,s)` | 통화 단위는 컬럼 comment에 명시 |
| 설명 / 비고 | `VARCHAR2` | 실제 최대 길이 기준 (100 / 200 / 4000 등) | 불필요하게 4000으로 통일하지 않는다 |
| 코드성 컬럼 | `VARCHAR2` | 코드 최대 길이 | 코드 값은 코드마스터 테이블로 관리 |
| 여부(Boolean) | `CHAR(1)` | 1 | `'Y'` / `'N'` 값만 허용, `_yn` 접미사 |
| 날짜 | `DATE` | - | 시각 정보가 필요 없는 경우 |
| 일시 | `TIMESTAMP` | - | 시각 정보가 필요한 경우 (`_at` 접미사) |

### 14.4 마이그레이션 파일 규칙 (백엔드 협의 참고)

- 마이그레이션 도구(Flyway 등) 사용 시 파일명: `V{버전}__{설명}.sql`
- 파일 상단에 아래 형식의 주석 헤더를 남긴다.

```sql
/*
 * 테이블/기능 : {테이블명 또는 기능명}
 * 작 성 일    : YYYY-MM-DD
 * 작 성 자    : {작성자}
 * 개    요    : {변경 목적}
 */
```

## 15. 메시지 및 로그 처리 표준

### 15.1 메시지 분류 기준

메시지는 노출 대상에 따라 2단계로 분리하여 관리한다.

| 단계 | 대상 | 내용 | 처리 위치 |
| --- | --- | --- | --- |
| 사용자 메시지 (Application level) | 최종 사용자 | 정상 안내, 입력값 검증 실패, 업무 처리 결과 등 | `features/{service}/messages.ts` + Toast/Alert |
| 시스템 메시지 (System level) | 개발자/운영자 | 서버 예외, 네트워크 오류, 인증 만료 등 | 공통 axios interceptor + 로거 |

- 사용자 메시지는 화면에 그대로 노출되므로 간결하고 경어체(`~습니다`, `~해주세요`)로 작성한다.
- 시스템 메시지(스택트레이스, 원본 에러 메시지 등)를 사용자 화면에 직접 노출하지 않는다.

### 15.2 메시지 코드 체계

```text
{서비스코드 3자리 대문자} + {일련번호 3자리}
```

- 서비스 코드는 3.2절 표를 따른다. (`PAT`, `RCP`, `LAB` 등)
- 일련번호는 001부터 서비스 내에서 순차 증가시킨다.
- 정의 위치: `features/{service}/messages.ts`

```ts
// features/labImaging/messages.ts
export const LAB_MESSAGES = {
  LAB001: '검사 접수가 생성되었습니다.',
  LAB002: '조회된 검사 오더가 없습니다.',
  LAB003: '필수 항목이 누락되어 접수를 처리할 수 없습니다.',
} as const;
```

- 백엔드 응답의 `message`가 코드 형태(`LAB001` 등)로 내려오는 경우, 프론트는 해당 딕셔너리로 변환 후 노출한다. 백엔드가 완성된 문구를 내려주는 경우 그대로 노출하되, 코드-문구 매핑은 백엔드 팀과 사전 협의한다.

### 15.3 프론트 메시지 처리 규칙

- 사용자 노출 메시지는 공통 Toast/Alert 컴포넌트로만 표시한다. (개별 `alert()` 사용 금지)
- 성공/실패에 따라 스타일을 구분한다. (성공: success, 실패: error, 안내: info)
- 폼 유효성 검증 메시지는 필드 하단에 인라인으로 표시하고, Toast는 서버 통신 결과에만 사용한다.

### 15.4 로그 처리

- `console.log`는 배포 브랜치(`develop`, `main`)에 커밋하지 않는다. (개발 중 임시 로그는 커밋 전 제거)
- 로그 출력은 공통 로거 유틸(`lib/logger.ts`)을 통해서만 수행한다.

```ts
// lib/logger.ts 사용 예시
logger.debug('[labImaging] 검사 오더 목록 조회', { params });
logger.warn('[labImaging] 재시도 발생', { retryCount });
logger.error('[labImaging] 접수 생성 실패', { error });
```

- 로그 레벨: `debug` / `info` / `warn` / `error` 4단계를 사용한다.
- 운영 환경에서는 `debug` 로그를 비활성화하고, `warn` 이상은 모니터링 도구(Sentry 등)로 전송한다.
- 레거시 방식인 "업무구분+일련번호" 로그 코드 체계 대신, 요청 단위 추적이 가능하도록 `traceId`(또는 Jira Key)를 로그 컨텍스트에 포함하는 것을 권장한다.

## 16. 주석 규칙

- 다른 사람이 바로 이해할 수 있도록 **필요한 주석을 작성**한다.
- 복잡한 비즈니스 로직, 조건 분기, API 연동 구간에는 주석을 남긴다.
- 주석은 "무엇을 하는지"보다 **"왜 이렇게 했는지"**를 중심으로 작성한다.
- 의미 없는 주석, 코드를 그대로 반복하는 주석은 작성하지 않는다.

**권장 주석 위치**

- 함수 / 컴포넌트 상단: 역할 설명
- Saga / API 연동: 요청 목적, 실패 처리
- 계산 / 상태 변환: 비즈니스 규칙
- 임시 처리 / 예외 로직: 사유와 TODO

**예시**

```ts
/**
 * 수술 스케줄 목록을 조회한다.
 * 수술실 필터가 있으면 scheduled_dt 기준으로 정렬한다.
 */
export function getSurgeryList(params: SurgeryListParams) {
  // 백엔드 응답 data만 사용 (code/message는 interceptor에서 처리)
  return surgeryApi.getList(params);
}
```

## 21. MSA 설계 원칙

### 21.1 서비스 책임 원칙

- 하나의 서비스는 하나의 업무(Bounded Context)만 책임진다.
- 서비스는 자신의 업무 데이터만 생성, 수정, 삭제할 수 있다.
- 다른 서비스의 업무를 대신 수행하지 않는다.
- 서비스 간 책임이 모호할 경우 업무의 소유자를 먼저 정의한다.

### 21.2 데이터 소유권 원칙

- 하나의 데이터는 하나의 서비스만 소유한다.
- 데이터의 생성, 수정, 삭제 권한은 소유 서비스만 가진다.
- 다른 서비스는 API 또는 Event를 통해서만 데이터를 조회하거나 요청한다.
- 다른 서비스의 데이터베이스를 직접 조회하거나 수정하지 않는다.

**예시**

| 데이터 | 소유 서비스 |
| --- | --- |
| 환자정보 | patient-service |
| 접수정보 | reception-service |
| 검사/영상 결과 | lab-imaging-service |
| 수술정보 | surgery-service |
| 수납정보 | billing-service |
| 공통코드 | admin-service |
| 문서양식 | admin-service |

> 14.1의 스냅샷 폐지 원칙(3.2)과 동일한 취지입니다: 타 서비스가 소유한 데이터는 참조 식별자만 보유하고, 표시 시점에 API로 조회합니다.

### 21.3 서비스 간 통신 원칙

- 서비스 간 직접 DB 접근을 금지한다.
- 서비스 간 데이터 교환은 REST API 또는 Event를 사용한다.
- 필요한 최소한의 데이터만 전달한다.
- 서비스 간 순환 의존성을 만들지 않는다.

### 21.4 공통 데이터 관리 원칙

- 공통으로 사용하는 데이터는 Admin Service에서 관리한다.
  - 공통코드
  - 병원 설정
  - 권한
  - 사용자 관리
  - 문서 양식(Template)
- 각 업무 서비스는 공통 데이터를 직접 수정하지 않는다.

### 21.5 문서 관리 원칙

기본 원칙은 다음과 같다.

- 환자가 작성한 종이문서는 병원의 문서관리 절차에 따라 보관한다.
- 시스템은 종이문서(전자문서 원본)를 저장하지 않는다.
- 시스템은 동의 여부, 동의일시 등 업무 정보만 관리한다.
- 빈 문서 양식(PDF)은 Admin Service에서 통합 관리한다.
- 각 업무 서비스는 필요한 문서를 출력만 수행한다.

**세부 결정 사항** (2026-07-11 "병원 문서관리 및 동의서 설계" 회의 결정)

- 전자문서 저장 여부: 프로젝트 범위에서는 전자문서를 관리하지 않는다. 실제 병원은 법적 분쟁·조회 편의를 위해 전자문서도 보관하는 경우가 많지만, 이번 프로젝트는 구현 범위를 명확히 하기 위해 전자문서는 관리하지 않기로 결정했다.
- 종이 동의서 보관: 의무기록실(MR) 또는 병원 문서보관 절차에 따라 보관하며, 시스템에는 파일을 저장하지 않는다. 자료실에는 환자가 작성한 문서가 아니라 '빈 양식'만 관리한다.
- 빈 양식 관리 구조: 공통 문서관리(Admin Service)에서 관리하고, 각 서비스는 문서를 출력만 하며 동의 여부만 저장한다.
  - 예시 문서: 수술동의서, 마취동의서, 조영제동의서, 개인정보동의서, 입원약정서, 각종 신청서, 진단서 양식, 환자교육자료

**문서관리 테이블 예시 (DOCUMENT_TEMPLATE, admin-service 소유)**

| 컬럼 | 설명 |
| --- | --- |
| document_id | PK |
| document_name | 문서명 |
| document_category | 문서 카테고리 (아래 참고) |
| document_type | 문서 유형 |
| file_path | 출력용 PDF 파일 경로 |
| version | 버전 |
| active_yn | 사용 여부 |

문서 Category 예시: `CONSENT`, `FORM`, `CERTIFICATE`, `EDUCATION`, `NOTICE`

**각 서비스 역할**

| 서비스 | 역할 |
| --- | --- |
| admin-service | 문서양식 관리, 버전 관리, 출력용 PDF 관리 |
| surgery-service | 수술동의 여부, 동의일시, 확인자 |
| lab-imaging-service | 검사/영상 동의 여부 |
| reception-service | 입원약정 여부 등 |

즉, 문서(양식)는 admin-service가 관리하고, 업무 데이터(동의 여부)는 각 서비스가 관리한다.

**admin-service를 문서양식 소유자로 정한 이유**: 병원 공통 자원이고 모든 서비스에서 사용하며, 중복 관리를 방지하고 유지보수가 용이하며 MSA 책임 분리가 명확하기 때문이다. (검토했던 대안: patient-service, 신규 document-service — 이번 프로젝트 범위에서는 비추천)

**최종 결정 요약**

- 전자문서는 관리하지 않는다.
- 종이문서는 병원 문서보관 절차에 따라 보관한다.
- 시스템은 동의 여부만 저장한다.
- 빈 양식(PDF)은 admin-service의 문서양식관리에서 관리한다.
- 각 업무 서비스는 문서를 출력하고 동의 여부만 관리한다.

> lab-imaging-service의 CONSENT 테이블이 이 원칙에 따라 설계되어 있다(동의 여부만 저장, `document_template_id`로 admin-service의 `DOCUMENT_TEMPLATE.document_id`를 참조 — DB FK가 아닌 서비스 간 논리적 참조).

### 21.6 상태(State) 관리 원칙

- 병원 업무 데이터는 삭제보다 상태 변경을 우선한다.
  - 접수 → 취소 → 완료
  - 검사 → 진행중 → 완료 → 승인 → 전송
  - 수술 → 예정 → 진행중 → 완료
- 가능한 이력을 보존하는 방향으로 설계한다.

### 21.7 ERD 설계 원칙

테이블 관계는 다음 기준으로 판단한다.

1. 부모 없이 자식이 존재 가능한가?
2. 업무적으로 항상 하나인가?
3. 이력이 발생하는가?
4. 취소·재접수·재생성이 가능한가?
5. 동일 FK가 여러 번 저장될 수 있는가?

위 질문을 기준으로 1:1, 1:N 관계를 결정한다.

> ⚠️ **재검토 필요**: lab-imaging-service ERD의 `LAB_RECEPTION-LAB_SCHEDULE`, `IMAGE_RECEPTION-IMAGE_SCHEDULE`는 "재조정 시 기존 레코드 UPDATE, 이력 미보유"를 근거로 1:1로 확정했다. 그런데 이 기준(질문 3 "이력이 발생하는가?")과 21.6 "가능한 이력을 보존하는 방향으로 설계한다" 원칙을 함께 적용하면, 검사/영상 일정은 재조정이 실제로 발생하는 업무이므로 1:N(이력 보존)으로 다시 판단될 여지가 있다. 기존 결정을 유지할지, 이번 기준에 맞춰 1:N으로 변경할지 팀 확인이 필요하다.

### 21.8 API 설계 원칙

- 조회: `GET`
- 등록: `POST`
- 수정: `PUT` / `PATCH`
- 삭제: 상태 변경 권장
- 모든 API는 공통 Response 형식을 따른다.

### 21.9 MSA 설계 원칙 (총괄)

- 서비스는 자신의 DB만 소유한다.
- 타 서비스의 테이블을 직접 참조하지 않는다.
- 필요한 식별자(ID)만 저장하고 상세 정보는 API로 조회한다.
- 서비스 간 결합도를 최소화한다.
- 공통 기능은 Admin Service에서 관리한다.
